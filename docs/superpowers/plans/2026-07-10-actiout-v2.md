# ActiOut v2 — Backend & Behavior Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move actual logged work from per-item aggregates to per-set rows, rebuild analytics on sets, add a snapshot/rollback subsystem, and ship four new features (quick session, last-time prefill, backfill/edit-completed, rest timer) — with only minimal functional UI wiring, ahead of a later UI redesign.

**Architecture:** A fresh Dexie `version(2)` schema (wipe-and-reseed, no `.upgrade()` — the last free schema change) introduces a `sessionSets` table and a `snapshots` table, and removes three dead fields. The service layer stays the sole writer; item completion becomes derived, warmups are excluded from analytics, and every whole-DB destructive op takes a compressed local snapshot first. UI work is the minimum wiring to keep screens compiling and functional against the new model — zero polish.

**Tech Stack:** React 18 + TypeScript (strict) · Vite · Dexie/IndexedDB · Zustand (transient UI) · Vitest + `fake-indexeddb` (real IDB, no mocks).

## Global Constraints

Copied verbatim from `docs/superpowers/specs/2026-07-10-actiout-v2-design.md` (the locked spec) — every task's requirements implicitly include these:

- **NEVER run `git add` / `git commit` / `git push`.** The user commits personally (`.claude/settings.local.json` denies these). At each "Commit" step, print the exact command and continue; do not execute it. *(user-mandated, CLAUDE.md)*
- **Bare-minimum UI protocol.** Barebone functional controls only. Verify via `tsc` / `npm run build` / tests. No browser walkthroughs, no cosmetic rounds, no UI review cycles. A single functional-correctness review pass at most.
- **Service-layer TDD, no mocks.** Tests run against real `fake-indexeddb`. No test re-derives its expected value by calling the implementation under test; expected values are written literally.
- **Per-row unit stamping (INV-5).** Every row storing a weight also stores its own `weightUnit`. Weight is **never** converted on write — conversion happens only at read/display time.
- **Wipe-and-reseed.** Schema declared fresh at `db.version(2)`, no `.upgrade()`. This is the **last** free schema change; a `formatVersion: 1` bundle is rejected.
- **Contiguous ordering (INV-2/INV-3).** `sequencePosition` (items in a session) and `setNumber` (sets in an item) are `1..n` with no gaps, renumbered transactionally on insert/remove/reorder.
- **Derived item completion (INV-4).** An item is "done" iff it has ≥1 set and all its **non-warmup** sets are `completed`. Never stored.
- **Warmups excluded from analytics (INV-7).** `isWarmup` sets are stored and shown in history but excluded from volume, PRs, and sequence stats.
- **Transactional lifecycle events (INV-8).** An `appEvents` row is written inside the same transaction as the mutation it records; a rolled-back mutation logs nothing; a no-op (deleting a non-existent id) logs nothing.
- **Definition of done, every task:** `npx tsc --noEmit` clean, `npx vitest run` all green, and at part boundaries `npm run build` clean.

**Delegation (tier tags per task):** the lead orchestrates and reviews; implementation is dispatched to subagents via `.claude/agents/actiout-implementer.md` (Sonnet default; Opus for the analytics rewrite; Haiku for boilerplate). Each dispatch gets a task-brief file, a report file, the "never git add/commit" override, and the TDD steps below.
